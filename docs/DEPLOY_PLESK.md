# План и подход к реализации сайта на хостинге с Plesk (без SSH)

Ограничения: хостинг без доступа по SSH, панель управления — **Plesk**. Ниже — варианты размещения и пошаговый план.

---

## 1. Что можно делать в Plesk без SSH

| Возможность | Обычно есть в Plesk | Комментарий |
|-------------|---------------------|-------------|
| Загрузка файлов | ✅ | Файловый менеджер или FTP/SFTP (учётная запись в Plesk) |
| Статический сайт (HTML/CSS/JS) | ✅ | Файлы в каталог домена (например `httpdocs` или `public_html`) |
| PHP | ✅ часто | Подходит для приёма форм без Node.js |
| Node.js | ⚠️ зависит от тарифа | В Plesk: «Расширения» → Node.js. Если есть — можно запустить наш backend |
| PostgreSQL | ⚠️ редко на shared | Чаще только MySQL. Уточнить в панели «Базы данных» |
| MySQL | ✅ часто | Можно хранить заказы в MySQL вместо PostgreSQL |
| Переменные окружения | ⚠️ | В Node.js-приложении в Plesk обычно задаются в настройках приложения |

**Итог:** статику (сайт) почти всегда можно разместить. Для сохранения заказов и CRM нужен либо Node.js + БД на хостинге, либо обходные варианты (PHP, внешние сервисы).

---

## 2. Подходы к реализации (три варианта)

### Вариант A. Только статика + формы через внешний сервис (проще всего)

**Суть:** на Plesk кладётся только статический сайт. Форма заказа не идёт на свой backend, а отправляет данные во внешний сервис.

**Плюсы:** работает на любом хостинге с Plesk, не нужны SSH, БД и Node.js.  
**Минусы:** заказы не в вашей БД и не в Twenty «из коробки»; CRM и отчёты — через сторонний сервис или вручную.

**Что сделать:**

1. Разместить на хостинге только файлы: `index.html`, `order.html`, `styles.css`, `script.js`, `order.js`, папку `data/` с `menu.json`.
2. Форму на `order.html` перевести на отправку, например:
   - в **Formspree** / **Getform** (получаете письма или доступ к заявкам в личном кабинете), или
   - в **Telegram-бот** (скрипт на сайте отправляет сообщение в ваш чат), или
   - оставить только кнопку «Заказать в WhatsApp» (данные не сохраняются на сайте).
3. Twenty CRM при желании использовать отдельно: заявки переносить вручную или настроить интеграцию позже (например, через Zapier/Make + Formspree → Twenty API).

**Когда выбирать:** хостинг без Node.js и без желания возиться с БД; важнее быстро запустить сайт.

---

### Вариант B. Статика + PHP-приёмник форм (без Node.js)

**Суть:** сайт тот же статический; форма отправляется на один PHP-скрипт, который сохраняет заказ в MySQL (если есть в Plesk) и/или отправляет в Telegram/email.

**Плюсы:** не нужен Node.js и PostgreSQL; используется то, что почти всегда есть в Plesk (PHP + MySQL).  
**Минусы:** нет прямой интеграции с Twenty из коробки (её можно добавить в PHP через вызов API Twenty).

**Что сделать:**

1. На хостинг загрузить статику (как в варианте A) + папку `api/` с файлом `order.php` (есть в репозитории).
2. В Plesk создать базу MySQL и пользователя; выполнить скрипт `backend/db/schema-mysql.sql` для создания таблицы `orders`.
3. В `api/order.php` прописать константы подключения к БД (DB_HOST, DB_NAME, DB_USER, DB_PASS).
4. Форма на сайте уже настроена: `order.js` сначала отправляет заявку на `/api/orders` (Node.js), при недоступности — на `/api/order.php`. При варианте B достаточно загрузить статику и `api/order.php`.
5. Опционально: в `order.php` задать TWENTY_API_URL и TWENTY_API_KEY — тогда заявки будут дублироваться в Twenty (создание Person).

**Когда выбирать:** в Plesk есть PHP и MySQL, Node.js нет или не хотите его использовать.

---

### Вариант C. Статика + Node.js backend на Plesk (как сейчас в проекте)

**Суть:** на хостинге поднимается наше приложение Node.js (Express), раздаёт статику и обрабатывает `POST /api/orders`; заказы пишутся в БД и при настройке уходят в Twenty.

**Плюсы:** полная логика как в текущем проекте, одна кодовая база, интеграция с Twenty из коробки.  
**Минусы:** нужны Node.js и БД (PostgreSQL или MySQL — тогда backend нужно адаптировать под MySQL).

**Что сделать:**

1. В Plesk: «Расширения» → установить/включить **Node.js** (если доступно на вашем тарифе).
2. Создать приложение Node.js: указать корень проекта (каталог с `backend/` и статикой) и точку входа, например `server.js` из `backend/`.
3. Задать переменные окружения в настройках приложения: `DATABASE_URL`, при необходимости `TWENTY_API_URL`, `TWENTY_API_KEY`, `TWENTY_PIPELINE_STAGE_ID`.
4. База данных: если есть только MySQL — добавить в проект поддержку MySQL (например, пакет `mysql2`) и таблицу по аналогии с `backend/db/schema.sql`; если есть PostgreSQL — использовать как сейчас.
5. Загрузить файлы по FTP/файловому менеджеру (или через Git, если Plesk даёт «Git» для домена).
6. В панели Node.js запустить приложение и привязать к домену (часто через проксирование на порт приложения).

**Когда выбирать:** в Plesk есть Node.js и подходящая БД; хотите сохранять заказы у себя и в Twenty без переписывания логики.

---

## 3. Где держать Twenty CRM

Twenty требует свой сервер (Docker/Node + PostgreSQL). На shared-хостинге с Plesk без SSH его не развернуть.

**Варианты:**

1. **Twenty Cloud** (если устраивает облачный вариант) — логин в CRM по их домену, интеграция с сайтом через API (наш backend или PHP вызывает их API).
2. **Отдельный VPS** — установить Twenty по официальной инструкции (Docker), открыть доступ по домену, на сайте (вариант B или C) при создании заказа вызывать API Twenty.

Сайт при этом живёт на Plesk; CRM — отдельно.

## 4. Пошаговый план внедрения на Plesk (общий)

### Этап 1: Подготовка (без хостинга)

1. Уточнить в панели Plesk:
   - есть ли **Node.js** и как создаётся приложение;
   - какие БД доступны (PostgreSQL, MySQL);
   - есть ли **PHP** и какая версия.
2. Выбрать подход: **A** (только статика + внешние формы), **B** (статику + PHP), **C** (статику + Node.js).
3. Для CRM: решить, где будет Twenty (облако или VPS), при вариантах B/C — завести API-ключ Twenty для интеграции.

### Этап 2: Размещение сайта (статическая часть)

1. Войти в Plesk → домен → «Файлы» (или подключиться по FTP).
2. Открыть корневой каталог сайта (часто `httpdocs` или `public_html`).
3. Загрузить:
   - `index.html`, `order.html`
   - `styles.css`, `script.js`, `order.js`
   - папку `data/` с файлом `menu.json`
4. Проверить: главная и страница заказа открываются, меню подгружается, ссылки и кнопки работают.

### Этап 3: Обработка заказов (в зависимости от выбранного варианта)

- **Вариант A:** поменять форму на `order.html` на отправку в Formspree/Telegram или оставить только WhatsApp; загрузить обновлённые файлы.
- **Вариант B:** загрузить `api/order.php`, создать БД и таблицу в MySQL, прописать в форме URL PHP-скрипта; при необходимости добавить вызов API Twenty в PHP.
- **Вариант C:** загрузить backend и статику, создать Node.js-приложение в Plesk, задать переменные окружения и строку подключения к БД, запустить приложение и проверить отправку формы.

### Этап 4: Домен, SSL, проверка

1. Убедиться, что домен привязан к сайту в Plesk.
2. Включить SSL (Let’s Encrypt в Plesk — обычно одна кнопка).
3. Проверить форму заказа: отправка уходит, заявки появляются (в письме, Telegram, БД или Twenty).

### Этап 5: CRM и доработки

1. При использовании Twenty — настроить объекты и pipeline по документу `docs/TWENTY_CRM_SETUP.md`.
2. Подставить реальные данные: WhatsApp, карта, при необходимости меню из `data/menu.json`.

---

## 5. Рекомендация

- **Если в Plesk нет Node.js или не хотите с ним работать:** начать с **варианта A** (статика + Formspree или Telegram), чтобы сайт сразу работал; позже при появлении PHP/MySQL можно перейти на **вариант B** и перенести приём заказов в свою БД и в Twenty.
- **Если Node.js в Plesk есть и есть БД:** ориентироваться на **вариант C**, сохраняя текущую архитектуру и интеграцию с Twenty; при наличии только MySQL — добавить в проект поддержку MySQL и один вариант схемы для неё.

В репозитории уже есть `api/order.php` и `backend/db/schema-mysql.sql` для варианта B.
